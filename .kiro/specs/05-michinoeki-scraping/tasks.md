# Implementation Plan

- [ ] 1. Foundation: 上流spec拡張とパッケージ雛形
- [x] 1.1 (P) 一覧ページの繰り返し要素から複数属性を一括取得する機能の追加
  - 既存のテキスト一括取得と対になる、指定した属性値をDOM順序のリストとして取得する任意取得手段を追加する
  - 要素自体や属性値が存在しない位置は欠損値として扱い、後続処理が座標欠落等を個別に判別できるようにする
  - 観測可能な完了条件: フィクスチャHTMLの複数要素から指定属性の値がDOM順のリストとして取得でき、要素・属性が欠落する位置には欠損値が返ることがテストで確認できる
  - _Requirements: 2.1, 2.2, 3.1_
  - _Boundary: HtmlPage(scraping)_

- [x] 1.2 (P) 施設情報への削除状態・最終確認日時の追加
  - 施設情報に有効/削除の状態と最終確認日時を保持する項目を追加し、出力前検証に状態の妥当性チェックを加える
  - 状態が既定(有効)の場合はJSON出力でキーを省略し、削除状態の場合のみ状態キーを出力するようにする
  - 観測可能な完了条件: 有効な施設のJSON出力に状態キーが含まれないこと、削除状態の施設では状態キーと最終確認日時が出力されること、不正な状態値が出力前検証で違反として検出されることがテストで確認できる
  - _Requirements: 8.1, 8.2, 8.3_
  - _Boundary: FacilityProperties(geojson)_

- [x] 1.3 前回出力済みGeoJSONの読み戻し機能の追加
  - 既存のGeoJSONファイルを読み込み、施設情報の列へ変換する機能を追加する(書き込み時の変換の逆方向)
  - 削除状態・最終確認日時を含む全項目を、書き込み時と矛盾しない形で復元する
  - 観測可能な完了条件: 書き込み済みのGeoJSONファイルを読み戻した結果が、書き込み前の施設情報の列(削除状態・最終確認日時を含む)と一致することがテストで確認できる
  - _Requirements: 8.1, 8.2, 8.4_
  - _Boundary: geojson reader_
  - _Depends: 1.2_

- [x] 1.4 (P) 道の駅スクレイパー用パッケージの雛形作成
  - 道の駅スクレイピング機能を配置する新規サブパッケージと、対応するテストパッケージの雛形を作成する
  - 観測可能な完了条件: 新設パッケージのimportが成功し、テストパッケージが空のテストスイートとして実行できる
  - _Boundary: michinoeki(パッケージ雛形)_

- [ ] 2. Core: 参照データ・範囲解決層
- [x] 2.1 (P) 実行対象範囲(全国・地方8区分・都道府県)の解決
  - 地方区分・都道府県のいずれも指定しない場合に全47都道府県を対象とし、地方区分指定時はその地方に属する都道府県のみ、都道府県指定時はその都道府県のみを対象として解決する処理を実装する
  - 地方区分と都道府県の同時指定、または存在しない値の指定はエラーとして扱う
  - 観測可能な完了条件: 指定なし・地方区分指定・都道府県指定・不正値指定・同時指定の各ケースで、それぞれ期待通りの都道府県列またはエラーが得られることがテストで確認できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4_
  - _Boundary: scope_

- [ ] 2.2 (P) サイト内都道府県コードとの対応・一覧ページURLの構築
  - 対象サイト固有の都道府県コード(公式コードとは異なる独自番号体系)の対応表を保持し、都道府県ごとの一覧ページURLを構築する処理を実装する
  - 観測可能な完了条件: 47都道府県すべてについて、公式コードから対応するサイト内コードを用いた一覧ページURLが正しく構築できることがテストで確認できる
  - _Requirements: 2.1_
  - _Boundary: site_urls_

- [ ] 3. Core: 収集層
- [ ] 3.1 (P) 一覧ページからの道の駅名称・詳細URL・座標の収集
  - 都道府県の一覧ページを取得し、道の駅ごとの名称・詳細ページURL・座標を抽出する処理を実装する
  - 一覧に該当要素が1件も無い場合は取得失敗として扱い、個々の要素で座標のみ解釈できない場合はその1件だけをスキップしつつ、スキップ件数と「一覧に実在した」という事実の両方を呼び出し側が参照できる形で返す
  - 観測可能な完了条件: フィクスチャHTMLから名称・詳細URL・座標が正しく対応付けて抽出できること、要素0件で取得失敗となること、一部要素の座標欠落時にその1件のみがスキップされスキップ件数と存在確認済みURLの集合に反映されることがテストで確認できる
  - _Requirements: 2.1, 2.2, 2.3, 3.1, 3.4, 7.1_
  - _Boundary: listing_
  - _Depends: 1.1, 2.2_

- [ ] 3.2 (P) 詳細ページからの付加情報抽出とプロパティへの変換
  - 詳細ページから名称・住所・郵便番号・電話番号・営業時間・駐車場台数・施設ホームページ・マップコード・施設設備タグを抽出し、道の駅向けのプロパティへ変換する処理を実装する
  - 名称が取得できない場合は構造変化として扱い、名称以外の項目は欠落時に個別に空値として扱い全体の抽出は継続する
  - 観測可能な完了条件: フィクスチャHTMLから郵便番号分離・駐車場台数(表記揺れを含む複数パターン)・施設設備タグ・ホームページ0/1/2件の各ケースが正しく抽出できること、名称欠落時に構造変化として扱われることがテストで確認できる
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 7.1_
  - _Boundary: detail_

- [ ] 4. (P) 前回出力とのマージによる削除状態遷移の実装
  - 前回出力済みの施設情報と今回のスクレイピング結果を対応付け、今回確認できた施設は状態を有効・確認日時を更新し、一覧からも確認できなくなった施設のみを削除状態へ遷移させる処理を実装する
  - 一覧には存在するが今回抽出できなかった施設は、削除判定の対象外として前回の状態のまま維持する
  - 削除状態のまま一定期間(1年)経過した施設は結果から完全に除外し、削除状態から再び確認できた施設は有効状態へ戻す
  - 観測可能な完了条件: 初回実行・消失による削除遷移・再出現による復帰・保持期間超過による完全除去・保持期間内での維持、および「一覧には実在するが今回抽出できなかった」施設が削除遷移せず前回状態を維持することの各ケースが、テストで確認できる
  - _Requirements: 8.1, 8.2, 8.3, 8.4, 8.5_
  - _Boundary: merge_
  - _Depends: 1.2, 1.3_

- [ ] 5. Integration: 都道府県単位・範囲単位のオーケストレーション
- [ ] 5.1 都道府県単位の収集ループと個別失敗のスキップ処理
  - 一覧取得結果を基に道の駅ごとに処理済み判定を行い、未処理の道の駅のみ詳細抽出を実行し、成功結果を蓄積する処理を実装する
  - 個々の道の駅の抽出失敗は都道府県全体を中断させず、スキップ件数を加算した上で警告ログを記録し、処理済みとして記録して次の道の駅へ進む
  - 観測可能な完了条件: 一部道の駅の詳細抽出が失敗しても他の道の駅の処理が継続され、成功結果の蓄積件数とスキップ件数がそれぞれ正しく確認できること、処理済み判定により重複処理が発生しないことがテストで確認できる
  - _Requirements: 4.1, 4.2, 4.3, 6.1, 6.2_
  - _Boundary: runner_
  - _Depends: 3.1, 3.2_

- [ ] 5.2 マージ・GeoJSON出力・index更新の結合
  - 都道府県分の収集結果と前回出力を統合し、検証を通過した場合のみGeoJSONファイルへ出力して管理ファイルの該当エントリを更新する処理を実装する
  - 一覧取得失敗・出力前検証違反時は当該都道府県の処理のみを中断してエラーを報告し、他の都道府県の処理には影響させない
  - 観測可能な完了条件: 1都道府県分の一覧取得から出力・管理ファイル更新までが一貫して動作すること、出力前検証違反時にファイルが書き込まれず当該都道府県のみが中断されることがテストで確認できる
  - _Requirements: 5.1, 5.2, 5.3, 8.1, 8.2, 8.3, 8.4, 8.5, 9.1, 9.2_
  - _Boundary: runner_
  - _Depends: 5.1, 4_

- [ ] 5.3 中断・再開をまたいだ都道府県単位の部分結果保持
  - 都道府県の処理途中で確定した収集結果・スキップ件数を、処理の完了を待たず逐次永続化し、都道府県の処理開始時に読み込んで復元する処理を追加する
  - 出力まで正常完了した場合にのみ、保持していた部分結果を消去する
  - 観測可能な完了条件: 都道府県処理の途中を模した中断の後に再実行した場合、中断前に成功していた収集結果とスキップ件数が失われずに最終出力へ反映されること、出力成功後は保持内容が消去され次回実行に影響しないことがテストで確認できる
  - _Requirements: 6.1, 6.2, 6.3, 4.3_
  - _Boundary: runner(部分結果キャッシュ)_
  - _Depends: 5.2_

- [ ] 5.4 範囲全体のオーケストレーションとレジューム完了処理
  - 解決済みの対象都道府県列を順に処理し、都道府県単位の失敗があっても範囲全体の処理を継続する処理を実装する
  - 指定範囲の全都道府県が処理完了した場合にのみ、レジューム状態全体をクリアする
  - 観測可能な完了条件: 一部都道府県が一覧取得失敗で中断しても他の都道府県の処理が継続されること、全都道府県が完了した場合にのみレジューム状態がクリアされ、失敗が残る場合はクリアされないことがテストで確認できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 6.3, 9.1, 9.2_
  - _Boundary: runner_
  - _Depends: 2.1, 5.3_

- [ ] 6. 実行エントリポイントの実装
  - コマンドラインから対象範囲(地方区分・都道府県)の指定を受け付け、範囲全体のオーケストレーションを起動する処理を実装し、実行コマンドとして登録する
  - 範囲指定が不正な場合はエラーメッセージを表示して非ゼロの終了コードを返し、この場合はいかなる通信も発生させない
  - 観測可能な完了条件: 登録した実行コマンドから範囲指定なし・地方区分指定・都道府県指定・不正指定の各ケースを起動でき、不正指定時はエラーメッセージと非ゼロ終了コードが返り通信が発生しないことが確認できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4_
  - _Boundary: cli_
  - _Depends: 2.1, 5.4_

  - 観測可能な完了条件: 結合テストが成功し、出力されたGeoJSONファイルと管理ファイルの内容が期待通りであることが確認できる

- [ ] 7.2 都道府県内での個別失敗の継続性と削除状態誤判定防止の結合検証
  - 都道府県内で一部道の駅の詳細抽出が失敗しても同一都道府県内の他の道の駅の処理が継続されること、抽出失敗した道の駅が前回出力で有効状態だった場合に削除状態へ誤って遷移しないことを検証する
  - 観測可能な完了条件: 抽出失敗が発生した道の駅を含む都道府県で、他の道の駅の出力が欠落しないこと、失敗した道の駅の状態が前回の有効状態のまま維持されることがテストで確認できる

  - _Depends: 5.2, 4_

- [ ] 7.3 レジューム(基本判定・中断復旧)の結合検証

  - 観測可能な完了条件: 処理済みURLが取得対象から除外されること、中断を模した状態からの再実行で最終出力に欠落が生じないこと、範囲完了後にレジューム状態が空になることがテストで確認できる

  - _Depends: 5.3, 5.4_

- [ ] 7.4 都道府県単位の失敗時継続性の結合検証
