# Implementation Plan

- [ ] 1. Foundation: パッケージ雛形・依存関係・共通例外・設定読み込み
- [x] 1.1 パッケージ雛形の作成と依存関係の追加
  - スクレイピング共通処理を配置するscrapingサブパッケージと、対応するテストパッケージの雛形を作成する
  - `pyproject.toml`の依存関係にHTTPクライアント(requests>=2.34)とHTMLパーサ(beautifulsoup4>=4.15)を追加する
  - `tech.md`のスクレイピング技術方針(BeautifulSoup/Scrapy使い分け検討中)を、requests+BeautifulSoup確定の内容に更新する
  - 観測可能な完了条件: 新設パッケージのimportが成功し、`pdm install`が新規依存を解決し、`tech.md`に確定した技術方針が反映されている

- [x] 1.2 (P) 構造化された例外階層の実装
  - エンジンの全失敗モードを表す基底例外と、取得失敗・パース不能コンテンツ・HTML構造変化それぞれに対応する専用例外を実装する
  - 各例外は対象URLや失敗箇所(ステータスコード・セレクタ等)を呼び出し側が参照できる属性として保持する
  - 観測可能な完了条件: 各専用例外が基底例外として捕捉でき、対象URL等の属性値を保持していることがテストで確認できる
  - _Requirements: 1.5, 2.6, 3.4, 4.1, 4.2, 4.4_
  - _Boundary: errors_

- [x] 1.3 (P) スクレイピング設定読み込みの実装
  - タイムアウト・最大リトライ回数・リトライ待機時間・最小リクエスト間隔を`pyproject.toml`の専用設定テーブルから読み込み、不変の設定値として提供する
  - 設定テーブル・ファイルが存在しない場合やキーの値が不正(負数・型不一致・構文エラー)な場合は、例外を送出せず該当項目を既定値に置き換えて警告ログを出す
  - 観測可能な完了条件: テーブル不在・キー省略・不正値それぞれのケースで設定値が既定値にフォールバックし、不正値ケースで警告ログが出力されることがテストで確認できる
  - _Requirements: 2.1, 2.7, 2.8_
  - _Boundary: ScrapingConfig_

- [ ] 2. Core: 取得・パース・レジュームコンポーネントの実装
- [x] 2.1 (P) レート制限・リトライ付きHTTP取得の実装
  - 送信前にリクエスト頻度制御を適用したうえで、設定されたタイムアウトでHTML・JSONコンテンツを取得する処理を実装する
  - サーバエラー・タイムアウト・接続エラーは設定された待機時間を空けて上限回数までリトライし、クライアントエラーは即時失敗として確定させる(リトライしない)
  - 取得成功時はレスポンスの文字エンコーディングを解決し、開始・成功・リトライ・最終失敗の各イベントを対象URL付きで共通ロガーに記録する
  - 観測可能な完了条件: 偽のHTTPセッションを用いたテストで、サーバエラー応答が上限回数までリトライされた後に取得失敗エラーを送出すること、クライアントエラーが即時失敗すること、全試行がリクエスト頻度制御を経由することが確認できる
  - _Requirements: 1.1, 1.2, 1.3, 1.4, 1.5, 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 5.1, 5.2_
  - _Boundary: PageFetcher_

- [x] 2.2 (P) HTML構造抽出の実装
  - 取得済みHTMLを解析し、セレクタ指定でテキスト・属性値を取得する任意取得手段(見つからなければ欠損値を返す)と、必須取得手段(見つからなければHTML構造変化エラーを送出する)を実装する
  - 必須取得では、要素自体が存在しない場合に加え、要素はあってもトリム後の値が空文字である場合も欠落として扱い、エラー送出前に警告ログを記録する
  - パースライブラリの実装詳細を呼び出し側に一切露出させないインタフェースとする
  - 観測可能な完了条件: フィクスチャHTMLに対し、任意取得の欠損時に欠損値が返ること、必須取得の欠落時(要素なし・空文字の両方)にHTML構造変化エラーと警告ログが確認できる
  - _Requirements: 3.1, 3.2, 3.3, 3.4, 4.1, 4.2, 4.3_
  - _Boundary: HtmlPage_

- [ ] 2.3 (P) URL単位のレジューム管理の実装
  - 処理済みURLの集合をレジューム機構に永続化し、処理済み判定・記録・クリアの手段を提供する
  - URLの記録操作の都度、状態全体を永続化し、中断がいつ起きても記録済みURLが失われないようにする
  - 観測可能な完了条件: あるURLを処理済みとして記録した後に新規インスタンスを再構築してもそのURLが処理済み判定されること、クリア後は判定が未処理に戻ることがテストで確認できる
  - _Requirements: 5.3, 5.4_
  - _Boundary: UrlResumeTracker_

- [ ] 2.4 抽出項目の宣言的な受け渡しの実装
  - 項目名・セレクタ・属性・必須有無からなる抽出項目の並びを受け取り、HTML構造抽出の任意/必須取得手段を適用して構造化されたレコードを生成する処理を実装する
  - 生成したレコードには取得元URLを対応付け、任意項目の欠損は欠損と判別可能な形でキーを保持したまま値のみ空にする。必須項目が1つでも欠落した場合はレコードを返さずHTML構造変化エラーを伝播させる
  - 項目名が重複した宣言は早期にエラーとして拒否し、全項目が欠損したレコードが生成された場合は対象URL付きで警告ログを出す
  - 観測可能な完了条件: 必須・任意が混在した抽出項目の並びから、全項目名をキーに持つレコードが生成されること、必須項目欠落時にHTML構造変化エラーが伝播すること、項目名重複時に早期エラーとなることがテストで確認できる
  - _Requirements: 4.1, 6.1, 6.2, 6.3_
  - _Boundary: extract_
  - _Depends: 2.2_

- [ ] 3. Validation: 結合フローと品質ゲートの検証

  - 観測可能な完了条件: フィクスチャ経由の結合テストが成功し、取得元URLが対応付けられた構造化レコードが得られることが確認できる

  - _Depends: 2.1, 2.2, 2.4_

- [ ] 3.2 リクエスト頻度制御とリトライ待機の重畳検証

  - 観測可能な完了条件: 短い間隔設定でのテストで、連続する取得試行の間隔が設定した最小間隔以上であることが確認できる
