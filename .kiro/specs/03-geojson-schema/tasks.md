# Implementation Plan

- [x] 1. Foundation: geojsonパッケージと都道府県対応表の整備
  - GeoJSONスキーマ関連モジュール群を配置するサブパッケージと、対応するテストパッケージの雛形を作成する
  - 47都道府県の番号(ゼロ埋め2桁)・小文字ローマ字名・日本語名の対応表を不変の参照データとして定義する
  - 番号から対応情報を引く参照手段を提供し、未知の番号は専用例外で拒否する
  - 観測可能な完了条件: 対応表が47件・番号昇順・重複なしであること、未知番号(`00`/`48`等)で専用例外が送出されることがテストで確認できる
  - _Requirements: 4.2, 4.3, 4.6_

- [x] 2. Core: データモデルとGeoJSON構造変換
- [x] 2.1 施設1件を表す不変のデータ型を定義する
  - 必須項目(施設名称、施設種別の2値、都道府県番号・都道府県名)とWGS84の座標(経度・緯度)を保持する
  - 共通の任意項目(住所、郵便番号、電話番号、営業時間、駐車場台数の内訳、ホームページURL列、情報源URL、施設設備タグ列)を定義する
  - SA/PA固有の任意項目(路線名、上り/下りの日本語2値列挙、方面)と道の駅固有の任意項目(マップコード)を同一の型に定義する
  - 観測可能な完了条件: 全項目を指定したインスタンスと必須項目のみのインスタンスの両方が構築でき、構築後の変更が禁止されていることがテストで確認できる
  - _Requirements: 1.4, 2.1, 2.2, 2.3, 2.4, 2.5, 2.6, 2.7, 2.8, 3.1_

- [x] 2.2 FeatureCollection構造への変換を実装する
  - 施設の列をRFC 7946準拠のFeatureCollection(各施設が`Point`ジオメトリを持つ1つのFeature)へ変換する
  - 座標は[経度, 緯度]の順序で出力し、`crs`メンバは出力しない
  - 値が無い任意項目はキーごと省略する
  - 観測可能な完了条件: 変換結果をJSONとして再読込した構造が、design.mdのデータ契約表(キー名・型・必須/任意)と一致することがテストで確認できる
  - _Requirements: 1.1, 1.2, 1.3, 2.9, 3.2_

- [x] 3. Core: 命名規則とスキーマ検証
- [x] 3.1 (P) 出力ファイル名の生成・解析を実装する
  - 都道府県と施設種別から命名規則に適合するファイル名(例: `01_hokkaido_michinoeki.geojson`)を生成する
  - ファイル名を解析して構成要素へ復元し、生成と解析の往復が一致する
  - 番号範囲外・ローマ字名と対応表の不整合・未知の施設種別・パターン不一致を専用例外で拒否する
  - 出力先ディレクトリの既定値を`geo-json/`として定義する
  - 観測可能な完了条件: 47都道府県×2種別の全組み合わせで生成→解析の往復一致がテストで確認でき、規則違反のファイル名が拒否される
  - _Requirements: 4.1, 4.2, 4.3, 4.4, 4.5_
  - _Boundary: naming_

- [x] 3.2 (P) Feature列のスキーマ検証を実装する
  - 必須文字列の非空(施設名称・都道府県名)、施設種別・上り/下り区分の列挙値、都道府県番号の実在と都道府県名との整合を検証する
  - 座標の値域(緯度±90・経度±180)を検証し、`NaN`/`inf`は値域違反として扱う
  - 違反を発見しても打ち切らず、全Feature・全項目を走査して違反箇所と理由のリストを返す(入力は変更しない)
  - 観測可能な完了条件: 複数の違反を含むFeature列で全違反が違反箇所付きで報告され、適合データでは空リストが返ることがテストで確認できる
  - _Requirements: 2.1, 2.3, 3.3, 5.2, 5.3_
  - _Boundary: validation_

- [x] 3.3 ファイル名検証とindex.json整合性検証を実装する
  - ファイル名の命名規則適合性を違反リスト形式で検証する(出力ゲートが利用する)
  - index.jsonの各エントリの`path`が命名規則に適合することを検証し、違反エントリを違反箇所付きで報告する
  - `updated_at`のISO 8601形式は既存のindex.json管理機能の日時型で保証されていることを整合性テストで確認する
  - 観測可能な完了条件: 命名規則違反の`path`を含むindex.jsonデータで違反が報告され、適合データでは空リストが返ることがテストで確認できる
  - _Requirements: 5.4, 6.1, 6.2, 6.3, 6.4_
  - _Depends: 3.1_

- [x] 4. Integration: 検証ゲート付き出力と統合検証
- [x] 4.1 検証ゲート付きGeoJSON出力を実装する
  - 書き込み前に必ずFeature列とファイル名の検証を実行し、違反が1件以上あれば全違反を保持した専用例外を送出してファイルを書き込まない
  - 検証合格時のみ、既存共通基盤のアトミック書き込みでFeatureCollectionをJSONとして出力し、出力先パスを返す
  - 検証違反(WARNING・違反件数と要約)と出力完了(INFO・パスと件数)を共通ロガーで記録する
  - 観測可能な完了条件: 違反ありデータで例外送出後に出力先へファイルが作成されていないこと、適合データで`geo-json/`配下にファイルが生成され再読込できることがテストで確認できる
  - _Requirements: 5.1, 5.5, 5.6_

- [x] 4.2 公開APIの整備とindex.json連携の統合検証
  - 型・変換・検証・出力の公開APIをパッケージの単一importパスから利用できるよう再公開する
  - 出力したファイル名を既存のindex.json管理機能で登録し、登録後のデータが整合性検証を通過する一連の流れを統合テストで確認する
  - 観測可能な完了条件: 出力→index登録→整合性検証の往復が一時ディレクトリ上の実ファイルで成功することがテストで確認できる
  - _Requirements: 6.1, 6.2, 6.3_
  - _Depends: 3.3, 4.1_

- [x] 4.3 品質ゲートの通過確認
  - 全テストスイートを実行し、新規モジュールのカバレッジがHTMLレポートに含まれることを確認する
  - lintを実行し、指摘0件で完了することを確認する
  - 観測可能な完了条件: `pdm run pytest`が全件成功しreport/にカバレッジが出力される、`pdm run ruff check .`が0件で終了する
  - _Requirements: 1.1, 2.1, 3.1, 4.1, 5.1, 6.1_
  - _Depends: 4.1, 4.2_
